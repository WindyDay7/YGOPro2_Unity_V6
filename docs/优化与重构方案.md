# 一、我先说当前项目“现在是什么样子”

我基于仓库内容看了一下, 当前项目大致是这样: 

## 1. 技术现状

- 仓库是一个 **Unity 项目**
- README 里最初是 **Unity 5.6.7** 项目, 但现在 `Packages/manifest.json` 已经出现: 
  - `com.unity.ugui`
  - 一堆较新的包
- 说明项目**历史很老, 但已经在某种程度上迁移/升级过 Unity 版本环境**
- 当前 UI 仍然明显带有: 
  - **NGUI**
  - 自定义窗口系统（`Servant`）
  - 自定义资源装配和模块切换逻辑

关键文件和目录: 

- `Assets/SibylSystem/Program.cs`
- `Assets/SibylSystem/Servant.cs`
- `Assets/SibylSystem/WindowServantSP.cs`
- `Assets/SibylSystem/Ocgcore/Ocgcore.cs`
- `Assets/SibylSystem/MonoHelpers/TcpHelper.cs`
- `Assets/YGOSharp/*`
- `Assets/ArtSystem/*`
- `Assets/NGUI/*`

项目文档里其实已经把现状说得很清楚了: 
**`Program` 是全局装配器, `Servant` 是窗口/模块框架, `Ocgcore` 是超重型对局核心.**

------

## 2. 当前架构优点

这个项目不是完全没架构, 相反, 它有几个非常好的基础: 

### 优点 1: 单场景 + 模块切换思路是对的

不是靠反复切 Scene 跑整个游戏, 而是在单场景内切换功能模块. 
这对卡牌客户端、桌面式窗口系统是合理的. 

### 优点 2: 对局表现层是复用的

联机、AI、残局、录像最终都汇入 `Ocgcore` 一套表现管线. 
这非常宝贵, 说明它其实已经具备“统一战斗流程”的思想. 

### 优点 3: 数据资源层相对独立

`YGOSharp` 负责卡片模型、卡组、禁卡表；资源管理器负责图片、字符串、Zip. 
这说明**领域模型和资源访问并没有完全烂在 UI 里**. 

### 优点 4: 自定义窗口系统说明作者当年就在解决“UI 模块化”

`Servant` 本质就是一套“窗口控制器 + 生命周期”的框架. 
只是它不是 MVVM, 也没有现代状态管理. 

------

## 3. 当前架构的核心问题

这个部分是你要重构的真正原因. 

## 问题 1: `Program` 是 God Object（上帝对象）

文档里说得很准: `Program` 同时负责: 

- 容器创建
- 摄像机
- 资源初始化
- 所有窗口创建
- 模块切换
- 每帧更新
- 输入上下文收集
- 延迟任务
- 生命周期收尾

这会带来几个后果: 

- 改一点初始化流程, 可能影响全局
- 很难做单元测试
- 很难做分阶段迁移
- 新模块会继续往 `Program` 上堆

------

## 问题 2: `Ocgcore.cs` 过重, 职责严重混杂

文档里提到它 9000+ 行, 这基本已经说明问题了. 
它同时在做: 

- 决斗状态机
- 场面模型
- 玩家输入选择
- 动画表现
- HUD
- 网络/本地消息解析
- 模式差异控制

这会导致: 

- UI 改动必须碰核心逻辑
- 对局逻辑很难复用到其他 View
- 很难引入自动化测试
- 回放 / AI / 联机虽然走同一条线, 但实现层面仍然耦得太死

------

## 问题 3: UI 框架是“控制器直连视图直连业务”

现在项目大概率是这种模式: 

- 业务模块直接拿 GameObject / prefab / NGUI 组件
- 业务类同时处理输入、显示、数据拼接、状态切换
- 窗口间靠全局引用和直接调用交互

这意味着: 

- UI 无法独立替换
- 状态流不清晰
- “谁改了这个界面”很难追踪
- 很容易出现一个按钮点击牵出一长串副作用

------

## 问题 4: 事件流是隐式的, 不可观察

比如联机流程现在大概是: 

- `TcpHelper` 收消息
- 推入队列
- 主线程处理
- 直接调 `Room` / `Ocgcore`

这是可运行的, 但不是一个现代可维护方案. 问题在于: 

- 没有统一消息边界
- 没有统一事件建模
- 不能方便记录、重放、调试、埋点
- 本地模拟和网络消息虽然“走同一路”, 但没有显式事件协议层

------

## 问题 5: 资源 / 配置 / 文件系统依赖比较硬编码

项目文档中也提到扫描这些目录: 

- `expansions/`
- `cdb/`
- `diy/`
- `data/`
- `pack/`
- `config/`

这类直接访问路径的方式在早期项目很常见, 但在重构阶段会出现问题: 

- 不利于迁移平台
- 不利于测试
- 不利于热更新/资源替换
- 不利于抽象配置源

------

## 问题 6: 历史遗留目录很多, 技术债较重

当前有: 

- `Assets/NGUI`
- `Assets/old`
- `Assets/modernUI`
- `Assets/transUI`
- `Assets/ArtSystem`
- 一堆旧目录和混杂资源

这类项目最怕的不是“写得差”, 而是**新旧体系混搭, 不知道哪一套是真正在线运行的主线**.

# 二、你要的目标架构: 我推荐什么

你说的是: 

- **MVVM**
- **事件 / 响应式（推荐）**
- **UI 用 UGUI**
- **重构整个项目**

我给你的推荐是: 

------

# 推荐总方案

## 核心架构

**MVVM + EventBus + Reactive State + UGUI + Presenter/Coordinator 辅助**

具体组合建议: 

### 1. View 层: UGUI

- 所有界面采用 `Canvas + RectTransform + Image + TextMeshPro + Button + ScrollRect`
- Prefab 化
- View 尽量只做显示和事件转发, 不做业务

### 2. ViewModel 层: MVVM 主体

- 每个界面一个 ViewModel
- 负责: 
  - 暴露状态
  - 接收用户意图
  - 调用 Application Service
  - 订阅领域事件 / 状态流
- 不直接依赖具体 UGUI 组件

### 3. 响应式层: 推荐用 Reactive

你说“事件 / 响应式（推荐）”, 我也确实推荐响应式优先. 
在 Unity/C# 里常见选择: 

- **R3**（新一代 Reactive 库, 作者 Cysharp）
- **UniRx**（老牌, 但维护活跃度和现代感不如 R3）
- **System.Reactive**（更通用, 但 Unity 实战不如前两者顺手）

### 我的建议: 

> **优先 R3, 如果团队已经熟 UniRx 也可以先 UniRx, 长期建议往 R3 走. **

你可以把响应式主要用在: 

- ViewModel 状态暴露
- 事件总线订阅
- UI 输入节流 / 防抖
- 异步加载状态
- 对局事件流投影到 UI 状态

------

## 4. 事件模型: 双层事件体系

我不建议“全项目只有一个事件总线”, 那样会变成另一种混乱. 
我建议分两类: 

### A. Domain Event（领域事件）

描述业务世界发生了什么, 例如: 

- `DuelStarted`
- `TurnChanged`
- `CardMoved`
- `LifePointChanged`
- `RoomPlayerJoined`
- `DeckSaved`

这些事件应当与 UI 无关. 

### B. UI Event / Navigation Event（界面事件）

描述界面行为, 例如: 

- `OpenSettingsPanel`
- `ShowToast`
- `ShowCardDetail`
- `NavigateToDeckEditor`
- `ShowDialog`

这类事件不进入领域层. 

------

## 5. 应用层: Application Service / UseCase

在 MVVM 和 Domain 中间加一层: 

- `StartDuelUseCase`
- `JoinServerUseCase`
- `LoadDeckListUseCase`
- `SaveDeckUseCase`
- `OpenReplayUseCase`

ViewModel 不直接和 `TcpHelper`、`Config`、`CardsManager` 打交道. 
它只依赖 UseCase / Facade / QueryService. 

这样做的好处是: 

- ViewModel 很薄
- 容易测试
- 以后换 UI、换资源层、换联机实现也比较稳

------

## 6. Domain 层: 纯业务模型

把真正和玩法有关的内容抽出来: 

- 卡组模型
- 房间模型
- 对局状态模型
- 卡片区域模型
- 玩家状态模型
- 回合/阶段状态机
- 指令/选择模型

Domain 层应该尽量: 

- 不依赖 UnityEngine
- 不依赖 UGUI
- 不依赖 NGUI
- 不依赖 GameObject

------

## 7. Infrastructure 层: 适配旧系统

这里是最关键的“平滑重构层”. 

你现在不可能一下把所有旧逻辑删掉, 所以要建立适配层: 

- `ITcpClientGateway`
- `ICardRepository`
- `ITextureRepository`
- `IZipArchiveService`
- `IConfigService`
- `IReplayService`
- `IPuzzleService`
- `IAiDuelService`

然后在初期用旧代码做实现: 

- `LegacyTcpHelperGateway : ITcpClientGateway`
- `LegacyCardsManagerRepository : ICardRepository`
- `LegacyConfigService : IConfigService`

这样你不是“推倒重写”, 而是“旧系统被新接口包起来”. 

------

# 三、推荐的技术选型

下面是我给你的更具体建议. 

------

## 1. Unity 版本建议

如果你真打算重构整个项目, **先统一 Unity 长期版本**, 别在旧版本和新包之间摇摆. 

建议: 

- **Unity 2022 LTS 或 2023 LTS**
- 不建议继续以 5.6 的历史兼容思路去做大重构

原因: 

- UGUI / TMP / Package 管理更成熟
- asmdef、测试框架、Addressables、Profiler 都更好用
- 更适合长期维护

------

## 2. UI 技术栈建议

### 必选

- **UGUI**
- **TextMeshPro**
- `CanvasScaler`
- `ScrollRect`
- Layout Group（慎用, 复杂性能热点处可手工布局）

### 可选增强

- DOTween: 做界面过渡和卡牌动画
- Localization 包: 如果你后面要现代化本地化
- Addressables: 如果未来要资源管理现代化

------

## 3. 响应式框架建议

### 推荐优先级

1. **R3**
2. UniRx
3. 自写轻量事件系统 + 普通 C# Observable 包装

### 为什么推荐响应式

因为你这个项目天然适合状态流: 

- 房间玩家列表变化
- 网络连接状态变化
- 卡组检索条件变化
- 卡片详情切换
- 回合/阶段变化
- 对局可操作按钮变化
- 异步卡图加载完成

这些都很适合: 

- `ReactiveProperty`
- `Observable`
- `Subject`
- `Command`

------

## 4. 依赖注入建议

强烈建议加 DI, 不然 `Program` 的问题会换个壳继续出现. 

可选: 

- **VContainer**: 我最推荐
- Zenject / Extenject: 也行, 但我更推荐 VContainer, 轻量现代

建议用法: 

- 根 LifetimeScope 放应用级服务
- 场景级 / 界面级 Scope 放局部服务
- ViewModel 通过构造函数注入依赖

------

## 5. asmdef 组织建议

这个项目一旦重构, 务必上 asmdef. 
建议至少拆成: 

- `Game.Domain`
- `Game.Application`
- `Game.Infrastructure`
- `Game.Presentation`
- `Game.Presentation.UGUI`
- `Game.LegacyAdapters`
- `Game.Editor`
- `Game.Tests.Unit`
- `Game.Tests.Integration`

这一步对大型重构帮助极大: 

- 限制引用方向
- 减少编译时间
- 强制架构边界

# 五、针对你这个项目, 具体应该怎么拆

下面进入最重要的部分: **按当前项目的真实结构, 怎么重构. **

------

## 1. 拆 `Program`

### 当前问题

`Program.cs` 是整个项目的大脑兼胃袋. 

### 重构目标

把它拆成多个独立服务: 

#### 拆分建议

- `AppBootstrapper`
  - 应用启动初始化
  - 依赖注入容器构建
- `GameLoopDispatcher`
  - 原来每帧驱动 servant / 网络 preFrame 的逻辑
- `SceneRoot`
  - 管理 Canvas 根节点、3D 根节点、UI 根节点
- `ModuleNavigator`
  - 模块切换 / 页面切换
- `TaskSchedulerService`
  - 代替 `go()` / `notGo()` 那类延迟执行
- `AppLifetimeService`
  - 退出、保存、释放资源

### 最终效果

`Program` 保留成一个很薄的 MonoBehaviour: 

- `Awake`: 启动容器
- `Start`: 调用 bootstrap
- `Update`: 只分发到 loop service

------

## 2. 拆 `Servant` 体系

### 当前问题

`Servant` 本质上是旧版“窗口控制器基类”. 

### 你的目标

你要做的是把 `Servant` 迁移为: 

- `Screen`
- `Window`
- `Popup`
- `Panel`

这几类 UGUI 视图对象 + 对应 ViewModel. 

### 替代设计

#### 原来: 

- `Servant.initialize()`
- `Servant.show()`
- `Servant.hide()`
- `Servant.preFrameFunction()`
- `ES_*`

#### 改成: 

- `IView.Bind(ViewModel vm)`
- `IView.Show()`
- `IView.Hide()`
- `IView.Dispose()`
- 具体输入走 UGUI 事件系统（Button/Toggle/EventTrigger/IPointerXXX）

### 迁移原则

不是先删 `Servant`, 而是: 

- 新界面不再继承 `Servant`
- 旧界面通过 `ServantAdapter` 暂时接入新导航系统
- 逐步把模块迁走

------

## 3. 拆 `Ocgcore`

这个是全项目最难的地方, 也是最值钱的地方. 

### 当前 `Ocgcore` 建议拆成 6 层

#### 1）DuelSession（对局会话）

负责: 

- 当前模式（联机/AI/录像/残局）
- 当前 duel 生命周期
- 当前参与者信息
- 当前状态快照

#### 2）DuelState（纯状态模型）

负责: 

- 场上卡片分布
- 玩家 LP
- 手牌/墓地/除外/额外
- 回合玩家
- 阶段
- 连锁信息
- 可选操作

注意: 
**DuelState 尽量纯 C#, 不要依赖 Unity. **

#### 3）DuelEventInterpreter（协议解释器）

负责把: 

- 网络消息
- Percy 消息
- 回放消息

解释成统一的领域事件, 例如: 

- `CardSummonedEvent`
- `CardMovedEvent`
- `SelectionRequestedEvent`
- `PhaseChangedEvent`

#### 4）DuelCommandService（玩家指令发送）

负责把用户操作变成: 

- 网络 Ctos 消息
- Percy 本地输入
- 回放控制命令

#### 5）DuelPresentationMapper（表现投影器）

负责把 DuelState / DomainEvent 映射成 ViewModel 需要的数据: 

- 手牌区域显示列表
- 场地区域卡片槽位
- 可点击按钮状态
- 连锁提示文本
- 阶段 HUD 状态

#### 6）DuelAnimationController（表现层）

负责 Unity 动画和特效: 

- 卡牌移动动画
- 召唤特效
- 攻击箭头
- 连锁动画
- 相机表现

### 关键思想

> **对局逻辑不直接驱动 Unity 对象, 而是先更新 DuelState, 再由 Presentation / Animation 层响应变化. **

这是整个项目现代化的关键. 

------

`TcpHelper` 同时干了:

# 六、MVVM 在这个项目里具体怎么落地

很多人说 MVVM, 但落地时会空泛. 我给你一套更贴近 Unity 的规则. 

------

## 1. View 只做什么

UGUI View 只负责: 

- 持有组件引用
- 绑定 ViewModel
- 监听按钮点击
- 订阅 ViewModel 的显示字段
- 播放局部动画/转场

不要在 View 里做: 

- 业务判断
- 卡组合法性计算
- 网络调用
- 文件读写
- 跨窗口导航策略

------

## 2. ViewModel 负责什么

ViewModel 负责: 

- 暴露响应式字段
- 处理用户意图
- 调用 UseCase
- 订阅领域事件
- 整理 UI 所需展示模型

例如: 

- `ReactiveProperty<string> Nickname`
- `ReadOnlyReactiveProperty<bool> CanJoin`
- `ReactiveCommand JoinCommand`

或者 R3 风格的 state/stream. 

------

## 3. Model 负责什么

注意这里的 Model 不要混成一个“大 Model”. 

应该分成: 

- Domain Model
- DTO / Query Model
- View Data Model

比如: 

- `Deck` 是 Domain Model
- `DeckSummaryDto` 是查询结果
- `DeckListItemVMState` 是 UI 展示模型

------

## 4. 响应式用到什么程度合适

不要把所有东西都 Rx 化. 
我建议下面这些适合响应式: 

### 很适合

- 输入框状态
- 按钮可用状态
- 当前选中卡片
- 房间玩家列表
- 连接状态
- 回合 / 阶段 / 连锁变化
- 卡图异步加载
- 搜索过滤条件联动

### 不建议强行响应式

- 很底层的纯算法
- 高频复杂动画细节
- 纯一次性初始化配置

原则: 

> **状态流和界面联动适合响应式；纯计算逻辑用普通 C# 更清晰. **

# 十一、你应该怎么优化“现在的项目”

这是从“工程治理”角度说. 

------

## 1. 先清边界, 不先清代码量

不要一上来就“删旧代码”“统一命名”“全改风格”. 
先做的是: 

- 明确哪些是 Domain
- 哪些是 UI
- 哪些是资源加载
- 哪些是网络
- 哪些是旧系统适配

边界清了, 代码自然好收敛. 

------

## 2. 先把依赖方向反过来

现在大概率是: 

- UI 依赖一切
- 业务类依赖 Unity
- 工具类依赖具体实现

要逐步变成: 

- Presentation -> Application -> Domain
- Infrastructure 实现接口, 反向注入

------

## 3. 先“加壳”, 再“换芯”

典型例子: 

### 旧: 

`DeckManager` 直接读 `CardsManager` / `Config` / 文件 / UI

### 新: 

`DeckEditorViewModel` -> `SaveDeckUseCase` -> `IDeckRepository` -> `LegacyDeckRepository`

这样你先把外壳建起来, 旧芯片还能继续用. 

------

## 4. UI 不要再直接操作数据源

以后所有 UGUI 界面都不要再直接: 

- 读文件
- 调静态单例
- 解析协议
- 改全局状态

全部经过 ViewModel / UseCase. 

------

## 5. 建立统一状态与日志

对大型重构项目非常重要: 

### 建议建立调试面板

- 当前 Screen
- 当前 Window 栈
- 当前网络状态
- 最近 50 条领域事件
- 当前 DuelState 摘要
- 当前资源加载状态

这会大幅提升你后续调试效率.

# 十四、风险与难点预判

你这个重构会遇到几个真实难点: 

## 1. `Ocgcore` 最难

不要一开始碰它. 
先建立外围新架构和方法论. 

## 2. 新旧 UI 并存期会比较长

很正常, 不要焦虑. 

## 3. 团队容易回到“图快直接写 MonoBehaviour”

必须靠代码评审和模板约束. 

## 4. 响应式容易滥用

尤其是: 

- 到处 Subject
- 到处 Subscribe
- 不做 Dispose
- 一切都 Observable 化

所以要规定: 

- 只在 ViewModel / 事件流 / 异步 UI 状态中使用
- 生命周期统一管理

## 5. 资源与路径系统可能卡你很久

这是老项目常见坑, 建议早抽象.
