# YGOProUnity_V2 整体框架说明

## 1. 项目定位

这个仓库是一个基于 Unity 5.6.7 的 YGOPro2 客户端实现。它不是把所有业务都塞进 Unity 场景脚本里，而是采用了“**Unity 负责运行时容器 + 自定义窗口系统负责交互 + OCG/Percy 负责规则执行 + YGOSharp 负责卡片数据访问**”的组合式结构。

从源码结构上看，这个项目最核心的目标是：

- 用 Unity 渲染 2D/3D UI 与对局表现；
- 兼容 YGOPro 通信协议；
- 同时支持联机、录像、残局、AI 对战几种运行模式；
- 通过 `ocgcore` 与 Percy 包装层统一承载决斗规则处理。

## 2. 技术栈与核心依赖

- **引擎层**：Unity 5.6.7，主场景是 [Assets/main.unity](../Assets/main.unity)
- **UI 框架**：NGUI，主要位于 [Assets/NGUI](../Assets/NGUI)
- **业务主代码**：集中在 [Assets/SibylSystem](../Assets/SibylSystem)
- **卡片/禁卡表/卡组数据模型**：集中在 [Assets/YGOSharp](../Assets/YGOSharp)
- **规则核心**：原生 `ocgcore` 动态库，通过 `coreWrapper` 与 `Ocgcore` 对接
- **AI/残局/回放桥接**：`Percy.smallYgopro` 与 `PrecyOcg`
- **资源读取**：本地目录 + Zip 包混合加载

## 3. 总体架构分层

可以把项目分成 5 层：

### 3.1 Unity 宿主层

入口是 [Assets/SibylSystem/Program.cs](../Assets/SibylSystem/Program.cs)。

`Program` 的职责非常重，基本相当于整个客户端的全局装配器：

- 持有绝大多数预制体、特效、UI 模块引用；
- 初始化相机体系与 UI 容器；
- 初始化配置、文本、卡片数据库、卡包与 Zip 资源；
- 创建所有业务窗口（项目中称为 `Servant`）；
- 驱动每帧更新、输入派发、延迟任务与网络消息分发；
- 负责模块切换、退出收尾、资源回收。

你可以把它理解为这个项目里的“应用程序主控器 + 服务注册表 + 运行循环”。

### 3.2 窗口/状态管理层

这一层由 `Servant` 体系承载：

- [Assets/SibylSystem/Servant.cs](../Assets/SibylSystem/Servant.cs)：最基础的窗口/模块基类
- [Assets/SibylSystem/WindowServantSP.cs](../Assets/SibylSystem/WindowServantSP.cs)：可弹出/缩放/记忆位置的窗口基类
- [Assets/SibylSystem/WindowServant2D.cs](../Assets/SibylSystem/WindowServant2D.cs)：纯 2D 窗口基类
- [Assets/SibylSystem/ServantWithCardDescription.cs](../Assets/SibylSystem/ServantWithCardDescription.cs)：自动联动卡片详情面板的业务窗口基类

这套机制的几个特点：

- 不依赖 Unity Scene 切换，而是在单场景内切换不同窗口显示状态；
- 所有业务窗口都通过 `Program.initializeALLservants()` 统一创建；
- `Program.shiftToServant()` 决定当前活跃模块，负责隐藏其它模块；
- `Servant.Update()` 内部统一处理鼠标/悬停/右键/滚轮等事件；
- 工具栏也是 `Servant` 自带能力，而不是每个窗口各写一套。

这说明项目整体更接近“桌面客户端式 UI 框架”，而不是典型的“场景驱动型 Unity 游戏”。

### 3.3 业务模式层

这一层是玩家真正可见的功能模块，主要包括：

- [Assets/SibylSystem/Menu/Menu.cs](../Assets/SibylSystem/Menu/Menu.cs)：主菜单入口
- [Assets/SibylSystem/selectServer/SelectServer.cs](../Assets/SibylSystem/selectServer/SelectServer.cs)：联机服务器选择/加入
- [Assets/SibylSystem/SelectDeck/selectDeck.cs](../Assets/SibylSystem/SelectDeck/selectDeck.cs)：卡组列表与管理入口
- [Assets/SibylSystem/deckManager/DeckManager.cs](../Assets/SibylSystem/deckManager/DeckManager.cs)：卡组编辑器、检索与换 Side
- [Assets/SibylSystem/Room/Room.cs](../Assets/SibylSystem/Room/Room.cs)：房间/观战/联机准备状态
- [Assets/SibylSystem/Ocgcore/Ocgcore.cs](../Assets/SibylSystem/Ocgcore/Ocgcore.cs)：真正的对局执行与表现层核心
- [Assets/SibylSystem/selectReplay/selectReplay.cs](../Assets/SibylSystem/selectReplay/selectReplay.cs)：录像浏览、转换、启动回放
- [Assets/SibylSystem/puzzleSystem/puzzleMode.cs](../Assets/SibylSystem/puzzleSystem/puzzleMode.cs)：残局模式
- [Assets/SibylSystem/Room/AIRoom.cs](../Assets/SibylSystem/Room/AIRoom.cs)：AI 模式入口
- [Assets/SibylSystem/Book/Book.cs](../Assets/SibylSystem/Book/Book.cs)：对局记录/情报窗口
- [Assets/SibylSystem/Setting/Setting.cs](../Assets/SibylSystem/Setting/Setting.cs)：设置面板

从职责上说：

- `Room` 管房间协议与房间态 UI；
- `Ocgcore` 管“场上发生了什么”；
- `DeckManager` 管卡组编辑和检索；
- `selectReplay` / `puzzleMode` / `AIRoom` 管不同玩法入口，但最终大都汇流到 `Ocgcore`。

### 3.4 数据与资源层

这一层由 `YGOSharp` 和若干资源管理器组成：

- [Assets/YGOSharp/CardsManager.cs](../Assets/YGOSharp/CardsManager.cs)：从 `.cdb` 读取 `datas` / `texts` 表并构建卡片对象
- [Assets/YGOSharp/Card.cs](../Assets/YGOSharp/Card.cs)：卡片数据模型
- [Assets/YGOSharp/BanlistManager.cs](../Assets/YGOSharp/BanlistManager.cs)：禁限卡表
- [Assets/YGOSharp/Deck.cs](../Assets/YGOSharp/Deck.cs)：卡组模型
- [Assets/SibylSystem/ResourceManagers/GameStringManager.cs](../Assets/SibylSystem/ResourceManagers/GameStringManager.cs)：系统文本、系列字段、卡片描述索引
- [Assets/SibylSystem/ResourceManagers/GameTextureManager.cs](../Assets/SibylSystem/ResourceManagers/GameTextureManager.cs)：卡图/UI 贴图异步加载与缓存
- [Assets/SibylSystem/ResourceManagers/GameZipManager.cs](../Assets/SibylSystem/ResourceManagers/GameZipManager.cs)：Zip 包登记与访问
- [Assets/SibylSystem/Config.cs](../Assets/SibylSystem/Config.cs)：本地配置键值存储
- [Assets/SibylSystem/InterString.cs](../Assets/SibylSystem/InterString.cs)：界面翻译文本入口

项目启动时会扫描这些目录：

- `expansions/`
- `cdb/`
- `diy/`
- `data/`
- `pack/`
- `config/`

其中：

- `.cdb` 提供卡片静态数据；
- `.conf` 提供系统文本与系列文本；
- `.ypk` / `.zip` 提供脚本、图片和扩展资源；
- `config/*.conf` 提供禁卡表、翻译和用户配置。

### 3.5 协议与规则执行层

这一层是项目最关键的“游戏逻辑中枢”：

- [Assets/SibylSystem/MonoHelpers/TcpHelper.cs](../Assets/SibylSystem/MonoHelpers/TcpHelper.cs)：YGOPro 网络协议的发送/接收分发器
- [Assets/SibylSystem/coreWrapper.cs](../Assets/SibylSystem/coreWrapper.cs)：对 `ocgcore` / Percy 相关结构和回放数据的 C# 包装
- [Assets/SibylSystem/precy.cs](../Assets/SibylSystem/precy.cs)：本地 AI / 残局 / 录像桥接层
- [Assets/SibylSystem/Ocgcore/Ocgcore.cs](../Assets/SibylSystem/Ocgcore/Ocgcore.cs)：把协议消息转成场面状态、动画、输入选择和 UI 反馈

这里面存在两条不同来源的“决斗输入流”：

1. **联机模式**：网络服务器 -> `TcpHelper` -> `Room` / `Ocgcore`
2. **本地模式**（残局 / AI / 录像）：`PrecyOcg` / Percy -> 伪装成协议消息 -> 同样送入 `TcpHelper` / `Ocgcore`

这也是整个工程最有意思的设计点：**不管消息来自网络还是本地模拟器，最终都尽量走同一套表现与状态更新通路。**

## 4. 启动流程

### 4.1 运行入口

Unity 启动后，`Program.Start()` 会完成以下动作：

1. 设置分辨率、帧率、鼠标特效对象；
2. 保存单例 `Program.instance`；
3. 调用 `initialize()` 分阶段初始化；
4. 延迟执行 `gameStart()`，显示背景与主菜单。

### 4.2 `initialize()` 的关键步骤

`Program.initialize()` 基本可以拆成两段：

#### 第一段：环境与相机

- 初始化头像缓存
- 初始化多个摄像机和 2D/3D 容器
- 创建背景模块

#### 第二段：数据与业务模块

- 读取翻译配置、普通配置
- 初始化贴图管理器
- 创建必要目录（如 `expansions/`、`replay/`）
- 扫描数据库、卡包、字符串、Zip 资源
- 初始化禁卡表
- 初始化所有 `Servant`
- 预加载常用特效与预制体
- 读取命令行参数并写入 `commamd.shell`

### 4.3 启动后的主循环

每帧 `Program.Update()` 会做三件事：

1. 收集鼠标、滚轮、悬停对象等输入上下文；
2. 遍历所有 `servants`，调用各自的 `Update()`；
3. 调用 `TcpHelper.preFrameFunction()`，把网络线程积压的协议包转为主线程业务回调。

因此这个项目真正的“主循环”不是藏在 `Ocgcore` 里，而是由 `Program` 驱动，`Servant` 负责消费。

## 5. 核心模块说明

## 5.1 `Program`：全局装配器

`Program` 有几个非常重要的角色：

- 资源注册表：大量预制体直接挂在它身上；
- UI 根节点创建者：维护 `ui_back_ground_2d`、`ui_main_2d`、`ui_windows_2d` 等容器；
- 模块调度器：统一切换 `menu`、`room`、`deckManager`、`ocgcore` 等模块；
- 延迟任务调度器：通过 `go()` / `notGo()` 实现简易定时执行；
- 应用生命周期管理者：保存设置、保存录像、释放 Zip 资源、回收 AI 进程。

如果后续要做大规模重构，`Program` 一定会是第一优先级，因为它目前承担了过多职责。

## 5.2 `Servant` 体系：项目自己的 UI 框架

`Servant` 的本质是“带生命周期和输入分发能力的窗口控制器”。

每个模块都通过这几个方法表达自己的行为：

- `initialize()`：创建 UI/注册事件
- `show()` / `hide()`：显示与隐藏
- `preFrameFunction()`：每帧逻辑
- `ES_*`：鼠标、悬停、右键等输入回调
- `applyShowArrangement()` / `applyHideArrangement()`：决定显示动画和位置布局

这个模式的好处是：

- 业务窗口之间接口统一；
- 不依赖 MonoBehaviour 继承树扩散到每个功能类；
- 大量逻辑可以写成“普通 C# 对象 + 必要时创建 GameObject”。

## 5.3 菜单与模式入口

`Menu` 是统一入口，主要负责：

- 打开设置、卡组、联机、录像、残局、AI 窗口；
- 检查更新；
- 轮询 `commamd.shell` 以支持命令行/外部调用；
- 退出时保存录像并强制结束进程。

`commamd.shell` 是一个比较老派但实用的桥接方式：

- 外部参数先被 `Program.readParams()` 转成文件命令；
- `Menu.checkCommend()` 再读取它，决定直接联机、编辑卡组、回放录像或运行残局。

这意味着项目支持“从外部启动并直接进入指定模式”的能力。

## 5.4 卡组系统

卡组相关功能分两层：

### 入口层：`selectDeck`

负责：

- 扫描 `deck/*.ydk`
- 排序、检索、重命名、复制、删除
- 选中某个卡组后进入编辑器

### 编辑层：`DeckManager`

负责：

- 快速搜索与高级搜索
- 根据 `YGOSharp.CardsManager` 结果构建卡片列表
- 编辑主卡组/额外/副卡组
- 按禁卡表约束保存
- 在换 Side 时复用同一套编辑 UI

因此 `DeckManager` 不只是“卡组编辑器”，它其实还是一个比较完整的卡片检索前端。

## 5.5 联机系统

联机流程可以概括为：

1. `SelectServer` 收集昵称、IP、端口、密码；
2. 启动线程调用 `TcpHelper.join()`；
3. `TcpHelper` 建立 `TcpClient`，发送 `PlayerInfo`、`JoinGame` 等协议包；
4. 接收线程把包塞进 `datas` 队列；
5. 主线程 `TcpHelper.preFrameFunction()` 解析 `StocMessage`；
6. 房间态消息交给 `Room`，对局态消息继续驱动 `Ocgcore`。

其中 `Room` 更偏“大厅/房间层”：

- 玩家进出
- 准备状态
- 聊天
- 观战人数
- Side 阶段
- 决斗开始/结束切换

而 `Ocgcore` 更偏“场内层”：

- 卡片对象管理
- 区域坐标计算
- 选择提示与可操作状态
- 动画、特效、血量、阶段、连锁显示
- 把协议消息变成视觉表现与输入请求

## 5.6 `Ocgcore`：整个项目的对局核心

[Assets/SibylSystem/Ocgcore/Ocgcore.cs](../Assets/SibylSystem/Ocgcore/Ocgcore.cs) 超过 9000 行，是这个仓库最核心也最重的文件。

它至少承担了这些职责：

- 维护场上所有 `gameCard` 实例；
- 计算各个区域的世界坐标与显示姿态；
- 管理不同模式（对战 / 观战 / 回放）；
- 接收协议消息并解析出具体的决斗步骤；
- 管理玩家名、LP、回合、阶段、连锁、按钮、可选项；
- 处理特效、召唤、攻击、移动、翻面等视觉表现；
- 响应玩家输入并回传到网络或 Percy。

从架构视角看，`Ocgcore` 更像：

- 一个“对局状态机”；
- 一个“场景表现控制器”；
- 一个“决斗输入适配器”；
- 再加上一部分“战斗 HUD 管理器”。

这也是为什么它是后续维护中最值得拆分的热点文件。

## 5.7 AI / 残局 / 回放

这三类模式并没有另起一套完全独立的表现层，而是通过 `PrecyOcg` 桥接到同一套对局界面。

### 残局

- 入口：[Assets/SibylSystem/puzzleSystem/puzzleMode.cs](../Assets/SibylSystem/puzzleSystem/puzzleMode.cs)
- 实现：读取 `puzzle/*.lua`，调用 `PrecyOcg.startPuzzle()`

### AI

- 入口：[Assets/SibylSystem/Room/AIRoom.cs](../Assets/SibylSystem/Room/AIRoom.cs)
- 实现：启动 `AI.Server.exe` 与 `WindBot/WindBot.exe`，然后通过本地 TCP 连接进入正常联机流程

### 回放

- 入口：[Assets/SibylSystem/selectReplay/selectReplay.cs](../Assets/SibylSystem/selectReplay/selectReplay.cs)
- 能力：
  - 浏览录像
  - 重命名/删除
  - `.yrp3d` 转 `.yrp`
  - 从录像反推出 `.ydk`
  - 启动回放并进入 `Ocgcore.Condition.record`

所以这三类模式本质上都在做一件事：**把不同来源的决斗数据灌进同一个场内执行器。**

## 6. 数据流说明

## 6.1 卡片数据流

1. `Program.initialize()` 扫描 `.cdb`
2. `YGOSharp.CardsManager.initialize()` 读取 SQLite
3. `Card` 对象进入内存字典 `_cards`
4. `DeckManager`、`Ocgcore`、卡片详情面板通过 `CardsManager.Get()` 读取卡片信息

## 6.2 文本数据流

1. `GameStringManager.initialize()` 读取 `.conf`
2. 把 `system`、`setname` 等文本保存到内存列表
3. UI 文案、效果描述、系列名通过 `GameStringManager.get()` 获取
4. `InterString` 和 `Config` 共同处理界面翻译与本地化配置

## 6.3 联机协议流

1. `SelectServer` 发起连接
2. `TcpHelper.join()` 建立 Socket 并启动接收线程
3. 接收线程调用 `SocketMaster.ReadPacket()` 读包
4. 包被推进 `datas`
5. 主线程解析 `StocMessage`
6. `Room` / `Ocgcore` 分别处理房间态与对局态
7. 玩家操作再通过 `TcpHelper.CtosMessage_*` 回发服务器

## 6.4 本地模式流

1. `PrecyOcg` 创建 Percy 引擎实例
2. Percy 通过 `receiveHandler` 把本地消息伪装成网络包
3. 仍然进入 `TcpHelper.addDateJumoLine()`
4. 后续流程与联机模式共用

这个统一入口设计极大降低了回放、残局和 AI 模式的额外 UI 成本。

## 9. 当前架构的优点与维护风险

### 优点

- 单场景切换，避免频繁场景装载；
- 不同玩法共用一套对局表现层，复用度高；
- `Servant` 基类让窗口交互模式一致；
- 卡片数据、文本、贴图、协议基本有明确分层；
- 支持扩展卡池、Zip 资源、外部命令启动，灵活性较高。

### 风险

- `Program` 和 `Ocgcore` 过于庞大，职责明显过载；
- 业务中存在较多全局单例与静态状态，测试和替换成本高；
- UI、协议、表现经常写在同一类里，模块边界不够硬；
- 有大量历史遗留目录与旧脚本，容易干扰新维护者判断；
- 文件系统依赖较强，对目录权限和相对路径比较敏感。

## 10. 如果后续要重构，建议优先做什么

建议按收益从高到低排序：

1. **拆 `Program`**：把资源注册、启动流程、Servant 切换、延迟任务拆出去
2. **拆 `Ocgcore`**：按“场面模型 / 输入选择 / 动画表现 / HUD / 协议处理”拆分
3. **抽网络事件总线**：让 `TcpHelper` 不直接硬编码到 `Room` / `Ocgcore`
4. **统一配置与路径访问**：减少散落的硬编码相对路径
5. **给模式入口建抽象层**：把联机、AI、残局、回放的“进入对局”过程统一接口化

## 11. 一句话总结

这个项目的本质不是“一个普通 Unity 卡牌游戏”，而是一个**以 Unity 为壳、以自定义窗口系统为骨架、以 `ocgcore`/Percy 为规则引擎、以 `YGOSharp` 为数据层的 YGO 客户端**。

如果你后续准备继续深入，我建议先重点啃懂三条主线：

- `Program` 的启动与调度
- `Servant` 的窗口框架
- `Room` / `TcpHelper` / `Ocgcore` 的消息流

理解了这三条线，整个仓库基本就能看透七八成。
