# 对局中卡片触发高亮与操作流程说明

本文档专门解释你在实际游玩时看到的这条流程:

> 一张卡先在卡片中上方自己出现一个像火花一样的闪光提示, 表示"这里现在有可操作项" → 鼠标移上去卡片会凸起、边框发光 → 卡片旁边弹出"发动效果 / 特殊召唤 / 攻击宣言"等可执行步骤 → 你再点击其中一个操作, 客户端把选择结果回传给规则核心或服务器. 

这条链路并不是某一个脚本单独完成的, 而是由以下几层代码协作完成的:

1. 网络/规则消息进入客户端
2. `Room` 把消息交给 `Ocgcore`
3. `Ocgcore` 解析当前这一步需要玩家做什么选择
4. `Ocgcore` 给对应卡片挂上"可执行操作按钮"和"高亮装饰"
5. `gameCard` 在每帧检测鼠标是否悬停或点击到自己
6. 悬停时把按钮显示出来, 并做放大/闪光效果
7. 点击卡片或点击弹出的按钮后, `Ocgcore` 组装回应包并发回
8. 回应后清理本轮选择状态, 等待下一条消息

如果你把它想成 MVC 风格, 大致可以这么理解:

- `TcpHelper` / `Room`:把"服务器说现在轮到你选了"送进来
- `Ocgcore`:决定"哪些卡可选、每张卡有哪些动作、需要显示什么提示"
- `gameCard` / `gameButton`:负责"这张卡怎么表现、鼠标移上去怎么弹按钮、点了之后怎么回调"
- `sendReturn()`:把玩家的实际选择发回规则系统

## 1. 先看你在游戏里看到的现象, 对应代码里是什么

你观察到的"触发点高亮 + 点卡后出现可执行步骤", 在代码里其实分成了三层表现:

### 1.1 卡片自己先出现的"火花提示"

这是你这次特别指出的那个现象:

- 鼠标还没有移上去
- 卡片中间偏上的位置, 已经自己出现一个火花/闪光点
- 它在提醒玩家:这张卡当前有操作或响应机会

这一层不是鼠标悬停效果, 而是 **规则消息到来后, 客户端主动给卡片挂上的状态提示**. 

它主要由两套机制一起完成:

- `gameCard.currentFlash`
- `gameCard.add_one_decoration(...)`

你可以把它理解成:

> 客户端先告诉你"这张卡现在有东西可点", 然后你才会把鼠标移上去看具体能做什么. 

### 1.2 卡片本体的"悬停强调提示"

这层通常表现为:

- 鼠标移上去后, 卡片边框额外发光
- 卡片会向镜头抬起一点
- 如果有叠放素材, 还可能顺手展开给你看

对应代码主要在:

- `Assets/SibylSystem/Ocgcore/Ocgcore.cs`
- `Assets/SibylSystem/Ocgcore/OCGobjects/gameCard.cs`

关键字段和方法:

- `gameCard.currentFlash`
- `gameCard.flashType.Active`
- `gameCard.flashType.Select`
- `gameCard.flashType.SpSummon`
- `gameCard.add_one_decoration(...)`
- `gameCard.RefreshFunction_decoration()`
- `gameCard.flash_line_on()`
- `gameCard.ES_enter_excited()`

### 1.3 点击卡片后弹出来的"可执行步骤按钮"

这层表现为:

- `发动效果`
- `特殊召唤`
- `通常召唤`
- `攻击宣言`
- `表示形式`
- `前场放置`
- `后场放置`

这些不是写死在 Unity Inspector 里的按钮, 而是 **运行时动态创建的 `gameButton`**. 

对应代码主要在:

- `Assets/SibylSystem/Ocgcore/Ocgcore.cs`
- `Assets/SibylSystem/Ocgcore/OCGobjects/gameButton.cs`
- `Assets/SibylSystem/Ocgcore/OCGobjects/gameCard.cs`

## 2. 整条链路的最上游:规则消息怎么进来

### 第一步:网络线程收到服务器消息

入口在:

- `Assets/SibylSystem/MonoHelpers/TcpHelper.cs`

接收线程里调用:

- `SocketMaster.ReadPacket(networkStream)`
- `addDateJumoLine(data)`

数据先进入 `TcpHelper.datas` 队列. 

### 第二步:主线程每帧分发消息

`Program.Update()` 每帧会调用:

- `TcpHelper.preFrameFunction()`

在 `TcpHelper.preFrameFunction()` 里, 如果收到的是:

- `StocMessage.GameMsg`

就会转发给:

- `Program.I().room.StocMessage_GameMsg(r)`

### 第三步:`Room` 把对局消息交给 `Ocgcore`

在:

- `Assets/SibylSystem/Room/Room.cs`

`StocMessage_GameMsg(BinaryReader r)` 会:

1. 调用 `showOcgcore()`, 确保对局界面显示出来
2. 生成一个 `Package`
3. `p.Fuction = r.ReadByte()`, 也就是当前的 `GameMessage`
4. `p.Data = new BinaryMaster(r.ReadToEnd())`
5. 调用 `Program.I().ocgcore.addPackage(p)`

也就是说, 这时消息还没真正执行, 只是进了 `Ocgcore` 的待处理队列. 

---

## 3. `Ocgcore` 怎么处理这些消息

### 3.1 `Package` 进入 `Ocgcore`

在:

- `Assets/SibylSystem/Ocgcore/Ocgcore.cs`

`addPackage(Package p)` 会把消息放进:

- `Packages`
- `Packages_ALL`

### 3.2 `Ocgcore.preFrameFunction()` 每帧消费消息

`Ocgcore.preFrameFunction()` 里会调用一个很核心的方法:

- `sibyl()`

而 `sibyl()` 里会循环处理 `Packages`:

1. 取出当前消息 `currentPackage`
2. 设置 `currentMessage = (GameMessage)currentPackage.Fuction`
3. 调用:
   - `logicalizeMessage(Packages[0])`
   - `practicalizeMessage(Packages[0])`
4. 处理完后把这条消息从队列中移除

这一步非常关键, 因为:

- `logicalizeMessage()` 更偏"更新状态 / 记录当前处于哪种选择阶段"
- `practicalizeMessage()` 更偏"把当前这一步需要显示给玩家的内容真正摆出来"

你关心的"卡片高亮、点卡弹操作"主要就发生在 `practicalizeMessage()` 对各种 `GameMessage.Select*` 消息的处理里.

## 4. 以"主阶段一张卡可以发动效果/特殊召唤"为例, 完整看一遍

你问的现象, 最典型的入口就是:

- `GameMessage.SelectIdleCmd`
- `GameMessage.SelectBattleCmd`
- `GameMessage.SelectChain`
- `GameMessage.SelectEffectYn`

其中"卡上有触发点, 点开后出现步骤"最接近的, 是:

- `SelectIdleCmd`:主阶段空闲时, 这张卡能做哪些动作
- `SelectBattleCmd`:战斗阶段, 这张卡能做哪些动作
- `SelectChain`:这张卡能不能连锁、有哪些可选效果

下面先讲 `SelectIdleCmd`. 

---

## 5. `SelectIdleCmd`:客户端是怎么给某张卡挂"可执行操作"的

代码在:

- `Assets/SibylSystem/Ocgcore/Ocgcore.cs`

在 `practicalizeMessage()` 处理 `GameMessage.SelectIdleCmd` 时, 代码会按协议结构依次读取多种可执行命令. 

### 5.1 第一批:通常召唤

代码会循环读取一批卡:

- 读 `code`
- 读 `gps`
- `card = GCS_cardGet(gps, false)`
- 找到卡之后:
  - `card.set_code(code)`
  - 创建 `gameButton(((i << 16) + 0), "通常召唤", superButtonType.summon)`
  - `card.add_one_button(btn)`

意思是:

> 这张卡现在有一个可执行动作, 动作编号是 `((i << 16) + 0)`, 按钮文案是"通常召唤". 

注意这里 **并没有立刻把按钮显示出来**, 只是先把它挂到了这张卡的 `buttons` 列表里. 

### 5.2 第二批:特殊召唤

流程类似, 但多做了两件事:

- 给卡加一个按钮:`特殊召唤`
- 给卡加一个高亮装饰:`mod_ocgcore_decoration_spsummon`
- 还可能设置:
  - `card.currentFlash = gameCard.flashType.SpSummon`

也就是说, 这种情况下玩家会先看到:

- 卡片上有蓝色/特殊召唤风格的提示

然后鼠标移上去时, 才会看到按钮弹出来. 

### 5.3 第三批:表示形式 / 前场放置 / 后场放置

这些也是同样的模式:

- 先创建对应的 `gameButton`
- 再挂到 `card.buttons` 里

### 5.4 第四批:发动效果 / 灵摆发动

这部分最像你说的"有触发点". 

处理流程是:

1. 协议里给出这张卡和对应效果描述
2. 如果是普通发动效果:
   - 创建一个 `Effect` 对象
   - 记录:
     - `ptr`
     - `desc`
   - 存入 `card.effects`
3. 如果这张卡还没有"发动效果"按钮:
   - 创建 `gameButton(..., "发动效果", superButtonType.act)`
   - `btn.cookieCard = card`
   - `card.add_one_button(btn)`
4. 再给卡片本身加高亮装饰:
   - `card.add_one_decoration(mod_ocgcore_decoration_card_active, ...)`
5. 如果当前这张卡是隐藏状态, 还会:
   - `card.currentFlash = gameCard.flashType.Active`

这一步就形成了你看到的效果:

- 规则系统告诉客户端"这张卡现在可以发动"
- 客户端给这张卡挂一个"发动效果"动作
- 同时给卡加发光提示, 让玩家知道"这里能点"

这里要特别补充一点:

- **这个提示不是鼠标移上去后才出现的**
- 只要 `Ocgcore` 处理完当前这条选择消息, 卡片就已经会自己亮起来

所以你实际看到的顺序应该理解成:

1. `Ocgcore` 先把这张卡标记为可操作
2. 卡片中上方先出现火花/闪光
3. 你再把鼠标移上去
4. 这时卡片才进入悬停放大态, 并把操作按钮摆出来

## 6. `SelectBattleCmd`:战斗阶段的逻辑是同一套, 只是按钮类型不同

`SelectBattleCmd` 里做的事情本质一样, 只不过按钮可能是:

- `发动效果`
- `攻击宣言`
- `主要阶段`
- `结束回合`

例如:

- 可以发动效果时:仍然会创建 `gameButton("发动效果")`
- 可以攻击时:创建 `gameButton("攻击宣言")`
- 如果是战斗阶段切换按钮:则不是挂在卡上, 而是加到 `gameInfo` 的全局操作条里

所以你会发现这个项目有两类按钮:

1. **卡片旁边的按钮**:`gameButton`
2. **屏幕上方或侧边的全局按钮**:`gameInfo.addHashedButton(...)`

## 7. 高亮到底是怎么画出来的

卡片上的"高亮"并不是只有一种实现, 而是有三种并行机制:

### 7.1 `currentFlash`:控制卡片自己先亮起来的火花点

在:

- `Assets/SibylSystem/Ocgcore/OCGobjects/gameCard.cs`

`gameCard` 在构造时就创建了三套 `FlashingController`:

- `SpSummonFlash`
- `ActiveFlash`
- `SelectFlash`

分别对应:

- 特殊召唤提示
- 发动/可操作提示
- 选择提示

当 `currentFlash` 改变时, `gameCard` 内部会切换哪一套闪光被激活. 

你可以直接把这段理解成:

- `flashType.SpSummon` -> 打开 `SpSummonFlash`
- `flashType.Active` -> 打开 `ActiveFlash`
- `flashType.Select` -> 打开 `SelectFlash`
- `flashType.none` -> 全部关闭

也就是说, `currentFlash` 更像是 **卡片自己的"状态型灯光提示"**. 

这正对应你观察到的现象:

> 鼠标还没移过去, 卡片中上方已经有一个小火花在闪

因为这一层根本不依赖鼠标, 它只依赖当前卡片是否被 `Ocgcore` 标记成:

- 可发动
- 可选择
- 可特殊召唤

### 7.2 `add_one_decoration(...)`:在卡片上方挂一个特效对象

比如:

- `mod_ocgcore_decoration_card_active`
- `mod_ocgcore_decoration_card_selecting`
- `mod_ocgcore_decoration_chain_selecting`
- `mod_ocgcore_decoration_card_selected`

这些是独立的装饰对象, `gameCard` 会在 `RefreshFunction_decoration()` 里不断把它们的位置更新到卡面附近. 

所以你看到的"闪光点、光圈、卡上方小特效", 很多是这一层实现的. 

### 7.3 `flash_line_on()`:鼠标移上去时的边框高亮

在:

- `gameCard.flash_line_on()`

会创建一个:

- `mod_ocgcore_card_figure_line`

并把它挂到卡面上, 同时启用 `HighlightingEffect`. 

这更像是"鼠标悬停强调边框", 不是协议驱动的"这张卡能操作"的提示. 

所以你在视觉上会看到两种效果叠加:

1. 协议告诉你这张卡可操作, 于是它自己先带一个火花/光点/装饰
2. 鼠标移上去后, 又额外出现悬停边框和放大效果

这两者不是同一个效果, 也不是同一个触发时机:

- 第一层: **先天提示层**, 不用悬停
- 第二层: **悬停展开层**, 需要鼠标移上去

## 8. 为什么按钮不会一开始就全显示, 而是要鼠标移上去才出现

这是 `gameCard` 自己的悬停系统在控制. 

关键代码在:

- `Assets/SibylSystem/Ocgcore/OCGobjects/gameCard.cs`

### 8.1 `RefreshFunction_ES()` 每帧检查鼠标

`gameCard.show()` 时会把 `Update()` 注册到 `Ocgcore` 的更新列表. 

而 `Update()` 每帧会调用:

- `RefreshFunction_ES()`

这个函数会做几件事:

1. 检查鼠标当前是不是指向这张卡或这张卡的相关对象
2. 如果点击了这张卡, 就调用:
   - `Program.I().ocgcore.ES_cardClicked(this)`
3. 如果当前处于悬停态, 就持续刷新悬停表现
4. 如果刚进入悬停态, 就调用 `ES_enter_excited()`
5. 如果鼠标离开了, 就调用 `ES_exit_excited(true)`

### 8.2 `ES_mouse_check()` 是怎么判断"鼠标在这张卡上"的

它会检查多个对象:

- `gameObject_event_main`
- `gameObject_event_card_bed`
- 竖版显示对象
- 所有按钮的事件对象 `buttons[i].gameObjectEvent`

这意味着:

> 鼠标不只是停在卡本体上算"还在这张卡的交互域里", 停在卡旁边弹出的操作按钮上, 也算. 

这就是为什么按钮弹出来后, 你把鼠标移到按钮上, 它不会立刻消失. 

### 8.3 `ES_enter_excited()` 会做什么

进入悬停态时, 会:

- 清掉旧 `iTween`
- 如果是可浮动卡片, 就执行 `flash_line_on()`
- 把卡旋转到更容易看清的位置
- 调用 `showMeLeft(true)`, 在左边卡片详情面板显示这张卡
- 如果这张卡有超量素材, 还会把叠放卡临时摊开

### 8.4 `ES_excited_handler_button_shower()`:真正把按钮摆出来

这部分非常关键. 

它会根据卡片的显示形态, 计算一个屏幕坐标起点, 然后调用每个按钮的:

- `buttons[i].show(...)`

也就是说:

- 按钮其实早就挂在 `card.buttons` 里了
- 真正"显示出来", 是悬停态里才发生的

这就是为什么你看到的是:

- 平时卡上只是有个高亮点
- 真正的操作步骤, 要等你鼠标移上去 / 点击卡片后才会出现

## 9. `gameButton` 是怎么被动态创建出来的

代码在:

- `Assets/SibylSystem/Ocgcore/OCGobjects/gameButton.cs`

`gameButton` 只是一个逻辑对象, 它在第一次 `show(Vector3 v)` 时才真正创建 Unity 预制体:

1. `create(Program.I().new_ui_superButton, ...)`
2. 拿到真正接收点击的事件对象:
   - `UIHelper.getRealEventGameObject(gameObject)`
3. 注册点击事件:
   - `UIHelper.registEvent(gameObject, clicked)`
4. 设置按钮图标和文字:
   - `iconSetForButton.setTexture(type)`
   - `iconSetForButton.setText(hint)`
5. 用 `iTween.ScaleTo` 做一个弹出动画

所以按钮不是预先摆在场上隐藏起来, 而是 **懒创建 + 悬停时显示**. 

这也正好解释了你观察到的完整体验顺序:

1. 先看到卡上有一个火花提示你"这里有操作"
2. 然后你才会把鼠标移过去
3. 这时按钮才真正从卡片边上弹出来

## 10. 点击卡片后, 为什么有时候直接执行, 有时候会再弹一个小选择框

因为卡片点击分两种情况:

### 情况 A:卡上只有一个动作

例如:

- 这张卡只有一个"特殊召唤"
- 或者只有一个"发动效果"

那点击操作按钮后, 通常就直接把这个动作编号发回去. 

### 情况 B:这张卡有多个效果描述

例如一张卡同时有多个可发动效果. 

这时 `ES_gameButtonClicked(gameButton btn)` 会判断:

- `btn.hint == "发动效果"`
- `btn.cookieCard.effects.Count > 1`

如果有多个效果:

1. 把每个 `Effect.desc` 转成一个 `messageSystemValue`
2. 调用:
   - `RMSshow_singleChoice("return", values)`
3. 玩家在弹窗里再选具体是哪一个效果
4. 最终在 `ES_RMS("return", result)` 里把 `ptr` 发回去

所以流程就变成:

- 卡片可发动
- 鼠标移上去
- 出现"发动效果"按钮
- 点"发动效果"后
- 如果这张卡只有一个效果:直接发包
- 如果有多个效果:再弹一个"选择具体哪个效果"的小弹窗

这也是你实际玩的时候会感受到"有些卡点一下就发动, 有些卡点开以后还要再选一层"的原因.

## 11. 点击卡片本体后, 到底谁来处理

入口在:

- `Ocgcore.ES_cardClicked(gameCard card)`

这个函数会根据 `currentMessage` 的不同, 执行完全不同的逻辑. 

### 11.1 在 `SelectCard / SelectTribute / SelectSum` 中

如果这张卡 `forSelect == true`:

- 它不是弹出动作按钮, 而是当作"选择目标卡"
- 会把它加入或移出 `cardsSelected`
- 然后调用 `realizeCardsForSelect()` 重新渲染所有可选卡

这种情况更像:

- "请选择素材"
- "请选择要解放的卡"
- "请选择若干张卡"

这里卡上的高亮通常是橙色 `Select`, 不是绿色 `Active`. 

### 11.2 在 `SelectChain` 中

如果 `card.forSelect == true`, 而且 `card.effects` 里有一个或多个可连锁效果:

- 只有一个效果:直接发回 `ptr`
- 多个效果:再弹选择框让你选具体效果

这就是连锁时最典型的"卡上冒出可点击提示"的实现. 

### 11.3 在 `AnnounceCard` 中

点击卡不是执行效果, 而是把卡号作为返回值发回去. 

也就是说, `ES_cardClicked()` 是一个 **按当前协议消息类型分流的总入口**.

## 12. 点击按钮后, 到底怎么发回服务器

### 12.1 卡片旁边按钮:`ES_gameButtonClicked`

在:

- `Ocgcore.ES_gameButtonClicked(gameButton btn)`

如果当前是:

- `SelectIdleCmd`
- `SelectBattleCmd`

通常会:

1. 取出 `btn.response`
2. 写入 `BinaryMaster`
3. 调用:
   - `sendReturn(p.get())`

如果是"发动效果"并且有多个效果, 则会先弹二级选择框, 最终还是走 `sendReturn(...)`. 

### 12.2 顶部/侧边的全局按钮:`ES_gameUIbuttonClicked`

例如:

- 战斗阶段
- 结束回合
- 取消选择
- 完成选择

这些按钮来自 `gameInfo.addHashedButton(...)`, 点击后会走:

- `Ocgcore.ES_gameUIbuttonClicked(gameUIbutton btn)`

它和 `gameButton` 的区别是:

- `gameButton` 是挂在某张卡旁边
- `gameUIbutton` 是挂在对局 HUD 上

### 12.3 `sendReturn()` 最终做什么

`sendReturn(byte[] buffer)` 会:

1. `clearResponse()` 清理当前这一轮 UI 状态
2. 调用 `handler(buffer)`

而这个 `handler` 在:

- `Assets/SibylSystem/Room/Room.cs`

`Room.show()` 里会设置:

- `Program.I().ocgcore.handler = handler;`

也就是说, `Ocgcore` 不直接关心网络细节, 它只是把"我要回应这一轮选择"的二进制包交给外部 handler. 

从架构上看, 这是一种很重要的解耦:

- `Ocgcore` 负责 UI 和当前对局状态
- `Room` / 网络层负责把回应真正送出去

## 13. `forSelect`、`effects`、`buttons` 三者分别代表什么

这是理解整个流程最重要的三个字段. 

### 13.1 `forSelect`

表示:

> 这张卡当前是不是"选择消息"中的候选对象

它常见于:

- 选择素材
- 选择目标
- 选择要发动连锁的卡
- 选择要宣言的卡

### 13.2 `effects`

表示:

> 这张卡当前挂着哪些可执行的效果描述

每个 `Effect` 里通常有:

- `ptr`:真正发回规则系统的编号
- `desc`:显示给玩家看的效果文本
- `flag`:有时用于区分效果类型

### 13.3 `buttons`

表示:

> 当鼠标移到这张卡上时, 应该显示哪些操作按钮

例如:

- 发动效果
- 特殊召唤
- 通常召唤
- 攻击宣言
- 表示形式

所以你可以把三者理解成:

- `forSelect`:这张卡能不能被点成"选择目标"
- `effects`:这张卡具体有几种效果可执行
- `buttons`:鼠标移上去时, UI 要给玩家展示哪些操作入口

---

## 14. 当玩家完成选择后, 旧高亮和旧按钮怎么被清掉

这一步在:

- `Ocgcore.clearResponse()`

关键清理逻辑包括:

- `cards[i].remove_all_cookie_button()`:清掉卡片上的动态操作按钮
- `cards[i].show_number(0)`:清掉选择编号/计数
- `cards[i].del_all_decoration()`:清掉高亮装饰物
- `cards[i].currentFlash = none`:关闭闪光状态
- `cards[i].prefered = false`
- `cards[i].forSelect = false`
- `cards[i].effects.Clear()`:清掉当前这轮效果列表

这就意味着:

> 每一条选择消息结束后, 客户端都会把"这一轮临时创建出来的按钮、闪光、可选状态"清空, 避免污染下一轮逻辑. 

所以这些高亮和操作步骤, 本质上都是 **"当前消息的临时 UI 状态"**, 而不是卡片永久自带的属性.

## 15. `realizeCardsForSelect()`:多选流程为什么会一边点一边变

当当前消息属于:

- `SelectCard`
- `SelectTribute`
- `SelectSum`
- `SelectUnselect`

时, 点卡后一般不会立刻发包, 而是会先:

- 改 `cardsSelected`
- 调 `realizeCardsForSelect()`

这个函数会:

1. 把 `allCardsInSelectMessage` 全部重置
2. 重新计算当前哪些卡仍然可选
3. 如果开启了智能选择 `smartSelect_`, 可能会自动提交
4. 给当前可选卡重新挂 `card_selecting` 装饰
5. 给已选中的卡显示编号
6. 如果已经满足最小/最大选择条件, 则显示 `完成选择` 按钮或自动发包

所以你看到的现象会是:

- 点一张卡
- 它上面出现 `1`
- 其他卡的可选状态可能变化
- 再点第二张卡
- 可能出现 `完成选择`

这套逻辑和"卡片上弹动作按钮"同属于 `ES_cardClicked()`, 但它们是不同分支.

## 16. 你看到的"卡片上有触发点", 本质上是哪一类消息

实际对局里, 这种现象常见于以下几种 `GameMessage`:

### 16.1 `SelectIdleCmd`

主阶段常见:

- 通常召唤
- 特殊召唤
- 发动效果
- 盖放
- 改变表示形式

### 16.2 `SelectBattleCmd`

战斗阶段常见:

- 发动效果
- 攻击宣言
- 阶段切换

### 16.3 `SelectChain`

连锁询问阶段常见:

- 哪张卡可以连锁
- 这张卡的哪个效果可以连锁

### 16.4 `SelectEffectYn`

这类通常不是卡旁边弹按钮, 而是直接:

- 高亮这张卡
- 弹出一个 yes/no 类型的确认框

所以它不一定会完全符合"点卡后弹步骤"的样子, 但底层来源也是同一套选择消息系统.

## 17. 为什么这个实现方式很巧妙

这个实现有几个很有意思的设计点. 

### 17.1 卡片本身是"半自治"的

`Ocgcore` 不会直接替你把每张卡的按钮摆出来, 它只负责:

- 给卡挂动作
- 给卡挂高亮状态

真正"鼠标移上去后怎么弹按钮", 是 `gameCard` 自己在每帧里处理的. 

这样做的好处是:

- `Ocgcore` 更像"状态分发器"
- `gameCard` 更像"局部 UI 控件"

### 17.2 动作按钮不是预制死的, 而是按协议动态生成的

这意味着不同卡、不同阶段、不同场面下:

- 同一张卡可以弹完全不同的按钮
- 同一个按钮文案, 背后可以对应不同 `response`

这非常适合卡牌游戏这种"当前能做什么高度动态"的场景. 

### 17.3 协议选择和视觉表现解耦得还不错

- 规则消息决定"能选什么"
- 卡片对象决定"怎么表现给玩家"
- `sendReturn()` 决定"怎么把答案交回去"

虽然整个 `Ocgcore` 很大, 但这条链路本身的分工其实是清晰的.

## 19. 一句话总结这条链路

如果只用一句话概括:

> 服务器或规则核心先告诉 `Ocgcore`"当前这张卡能做什么", `Ocgcore` 再把这些动作挂到 `gameCard` 上并加高亮, `gameCard` 在鼠标悬停时把动作按钮动态显示出来, 最后点击卡片或按钮后由 `Ocgcore` 把对应编号发回去. 

也可以拆成更程序化的版本:

1. `TcpHelper` 收包
2. `Room.StocMessage_GameMsg()` 转成 `Package`
3. `Ocgcore.sibyl()` 调 `practicalizeMessage()`
4. `SelectIdleCmd/SelectBattleCmd/SelectChain` 给卡挂 `buttons + effects + decoration + currentFlash`
5. `gameCard.RefreshFunction_ES()` 检测鼠标悬停/点击
6. `ES_enter_excited()` 把卡片凸起并显示按钮
7. `gameButton.clicked()` 或 `ES_cardClicked()` 把操作交回 `Ocgcore`
8. `sendReturn()` 把选择送回规则系统
9. `clearResponse()` 清理旧高亮和旧按钮

## 20. 对 Unity 小白最实用的理解方式

你现在不用把 9000 行 `Ocgcore` 一下看完. 你只要先抓住这三个事实, 就已经理解了这条机制的 80%:

### 事实 1

**"能不能操作"不是卡自己决定的, 是规则消息决定的. **

### 事实 2

**卡片只是把 `Ocgcore` 给它的"操作列表"用 UI 的方式表现出来. **

### 事实 3

**高亮、弹按钮、点按钮发包, 都是"当前这一步消息"的临时状态, 不是永久状态. **

你只要理解这三点, 以后再看别的选择消息, 比如:

- 选素材
- 选目标
- 选位置
- 选连锁

你都会更容易读懂.
